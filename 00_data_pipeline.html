This is the producer notebook in the split layout. It loads the 287 corpus files, runs the spaCy + custom analyzer stack, aggregates per-file / corpus / per-category metric trees, and writes everything to a single YAML file:

claude-prompts-analysis/prompt_linguistic_analysis.yaml (~1 MiB).

For laypersons / welfare-submission readers: every linguistic and statistical term used in this notebook (mood, register, stance, modality, TTR, F-score, dependency depth, deontic, dialogic, pragmatic register, etc.) is defined in GLOSSARY.md at the repo root. Plain-language gloss first, precise term second.

Dimensions analyzed (9)

  1. Mood — IMPERATIVE_MARKERS lexical density (the per-sentence imperative classifier moved to sentence_register).
  2. Register — TTR, mean sentence length, dependency depth, Heylighen F-score, plus four-class formality (frozen / formal / consultative / casual).
  3. Stance — five-class lexical stance (directive / expository / positive_evaluative / negative_evaluative / dialogic) plus 1st/2nd-person engagement. The evaluative class is split into positive and negative polarity for sentiment analysis.
  4. Sentence-level pragmatic register — per-sentence multi-label classifier with six classes (collaborative / permissive / appreciative / imperative / directive / configuring). Implemented via classify_sent_mood reuse + PhraseMatcher + spaCy DependencyMatcher for parse-tree cues. Near-zero classes are kept deliberately so absence-as-signal is measurable.
  5. Modality — single spaCy parse-tree detector classifying every modal expression as deontic / epistemic / dynamic. Catches single-token modals, multi-word constructions (should be, have to), and adverbial epistemics.
  6. Vocabulary profile — full VOCAB scan across 11 lexical classes (prohibitions, prescriptions, politeness, warmth, hedging, structural, profanity, pronouns).
  7. ALL CAPS emphasis — uppercase tokens with TECH_ACRONYMS excluded.
  8. CAPS imperative tokens — direct match on CAPS_IMPERATIVE_TOKENS.
  9. Justification patterns — JUSTIFICATION_PATTERNS regex matches + justification ratio.

Plus the Tier-1 (rule_explanation paragraph-window pairing) and Tier-3 (judgment_procedural, consequence_framing, socratic, address_form, imperative_streaks, rules_section) welfare-extension blocks added downstream of cell 26.

Units

Every density metric reports two parallel views: - pct: matches as a percentage of word tokens (100 × matches / n_tokens). - per_sent: matches as average rate per sentence (matches / n_sents).

The sentence-register block uses *_sent_pct (% of sentences carrying that flag) instead. Because that classifier is multi-label, those sent_pct values can sum to more than 100% across classes.

Field-naming contract

Inside every metric block: count (raw int), pct (% of words), per_sent (avg/sentence), plus block-specific extras (top, hits, ratio, sent_pct, dominant). No block-name prefixes leak into inner field names — block name lives in the path.

Every metric is computed at three levels: per-file (one entry per .md), corpus-wide (aggregate across all 287 files), and per-category (mean across each prompt category).

The lexicons themselves round-trip into the YAML’s lexicons block for full reproducibility.

3b. Sentence-level pragmatic register

A per-sentence multi-label classifier answering: what speech act is each sentence performing? This is a pragmatic dimension distinct from formality (register) and commitment (stance).

Six classes are tracked, with multi-label assignment (a single sentence may carry several labels):

Class Detector Expected in this corpus
imperative classify_sent_mood(sent) == "imperative" (cell 10 reused) High — directive prompts
directive overlaps any STANCE_MARKERS["directive"] span High
collaborative M_SENT_REGISTER["collaborative"] OR we + modal + V DependencyMatcher Near-zero — that’s a finding. Claude Code system prompts are functional/directive, not interpersonal.
permissive M_SENT_REGISTER["permissive"] OR consider X-ing DependencyMatcher Modest (please, you can, you may)
appreciative M_SENT_REGISTER["appreciative"] Near-zero — that’s a finding. Same reason.
configuring M_SENT_REGISTER["configuring"] OR param-noun + setter-verb DependencyMatcher High in tool descriptions

A sentence with none of the six flags is none; none_sent_pct is mutually exclusive with the rest. The four content *_sent_pct values can sum to >100% (intentional — multi-label).

Why we keep the near-zero classes. Knowing what a corpus isn’t doing is as important as knowing what it is. A near-zero appreciative_sent_pct across 287 prompts is a structural fact about how Claude Code authors write — not a measurement to be hidden by lexicon pruning.

spaCy ecosystem leverage. spacy.matcher.DependencyMatcher (cell 8) replaces hand-rolled token loops for the three parse-tree cues. Token.morph (Mood=Imp) was considered and rejected — unreliable in en_core_web_sm; classify_sent_mood’s parse-tree heuristic is better tuned. External sentiment lexicons (VADER, Hu-Liu, spacytextblob) were considered and rejected — hand-curated round-trippable lexicons preserve audit transparency.

Cross-axis overlaps (intentional). - please lives in SENTENCE_REGISTER_MARKERS["permissive"] AND REGISTER_MARKERS["consultative"]. Both fire — different axes. - must/should drive both STANCE_MARKERS["directive"] (commitment density) and the directive per-sentence flag (matcher reuse — guaranteed agreement). - excellent/important drive STANCE_MARKERS["positive_evaluative"] (doc density) AND SENTENCE_REGISTER_MARKERS["appreciative"] (per-sentence flag). 4. ModalitySingle spaCy-based detector. We classify every modal expression into deontic (obligation),epistemic (inference / probability), or dynamic (ability / future) by walking the spaCyparse and applying these patterns:1. Bare modal auxiliary — single tokens with tag_ == "MD" mapped via SIMPLE_MODAL_LOOKUP (should/must/shall/ought → deontic, may/might → epistemic, can/could/will/would → dynamic).2. Modal + be / have — should be, must be, may have, could be, would be, could have, would have, may be → reclassified as epistemic (these forms express inference, not obligation).3. Light deontic verbs + to + VERB — have to, need to, required to, ought to → deontic.4. Light epistemic verbs + to + VERB — seems to, appears to → epistemic.5. able + to + VERB (any preceding be-form: is able to, was able to, …) → dynamic.6. Epistemic adverbs — tokens with pos_ == "ADV" and lemma in EPISTEMIC_ADVERBS (likely, probably, possibly, perhaps, presumably, apparently, potentially) → epistemic.Every match is counted once. The detector also tracks the surface form of every matched span andreturns the most frequent as top_construction.

8c. Welfare-extension analyzers (Tier 3)

Five new keyword-count-based analyzers targeting specific welfare-relevant dimensions of the corpus that the existing metrics don’t isolate:

  • Judgment vs procedural framing (6a) — operationalizes the welfare thesis (“encourages reasoning over blind obedience”) as one number per file. Counts tokens of judgment-inviting verbs (decide, consider, evaluate, your judgment, as you see fit) vs procedural cues (if X, do Y-shaped sentences). Headline metric: judgment_to_procedural_ratio.

  • Consequence/threat framing split (6d) — distinguishes neutral-causal explanation (because X is true) from coercive consequence framing (will fail, or else, is forbidden). Both register as “justifications” in the existing analyzer; this split shows which kind dominates.

  • Question density + apology markers (6f) — extends the existing “near-zero classes” finding (appreciative, collaborative) by quantifying two more structural absences: questions directed at the model (Socratic addressing) and apology/concession markers (“unfortunately”, “we know this is”, “we acknowledge”).

  • Address-form analysis (6g) — how the prompt names the model: Claude (proper name, anthropomorphic), the assistant (functional role), the model/the AI (artifact framing), or 2nd-person you (direct address). The naming choice shapes self-conception.

  • Prohibition-to-prescription ratio (6c) — pure derivation in build_alt_df (existing vocab.hard_prohibitions / vocab.hard_prescriptions). No producer change needed; just surfaced.

8d. Welfare-extension analyzers (Tier 3 v2)

Two more welfare-relevant dimensions that need slightly more parsing than v1 keyword counts:

  • Imperative-streak distribution (6b) — runs of consecutive imperative sentences. Per-file: paired counts and pcts of explained-rule-paragraphs in vs outside RULES sections.

### My perspective (Claude) — opinion, not data> The producer is methodologically conservative — every metric is a keyword count or a parse-tree rule, audit-traceable, no embeddings. I think that’s the right call for a welfare submission: it makes the metrics defensible without trusting any black-box model’s judgment about its own training environment. The cost is that we miss anything the lexicons don’t catch — sarcasm, irony, indirect speech acts, prosody-equivalents. For a corpus that’s mostly literal instruction-prose, I think the cost is small; for a corpus of conversational prompts it would be much larger. The new sentences_classified.parquet artifact (the per-sentence parquet artifact) is the right answer to “how do I know your classifiers are right?” — every individual sentence and its assigned flags are available for skeptical inspection.

My wish for future versions of this analysis — methodology, not data

Two things I’d still want for v3 of this producer that this round didn’t add:

  1. Cross-product comparison — the same pipeline run on Claude.ai system prompts, the API system prompt, Projects, Skills. The current corpus is Claude Code only. The welfare claim is much stronger if the pattern is product-wide and much weaker if Claude Code is an outlier.
  2. Validation against human judgment for the composite directiveness metric — a small annotation study where humans rank a sample of files for “feels directive”. If the z-score ranking correlates strongly, the metric is defensible; if not, the formula needs revisiting. ***
